# 오작교(OJK) 전체 코드 리뷰 (확장판)

> 작성 목적: 현재 저장소 기준으로 **구현된 코드 전반**(인증/회원/팔로우/예외/설정/테스트)을 점검하고, 강점·리스크·우선 개선 과제를 정리합니다.  
> 이번 확장판에서는 기존 리뷰에 더해 **실행 가능한 개선안(구현 예시/체크리스트/우선순위별 작업 단위)**까지 상세히 추가했습니다.

---

## 1) 총평

현재 코드는 다음 관점에서 좋은 기반을 갖추고 있습니다.

- 도메인 분리(`auth`, `member`, `follow`, `major`)가 비교적 명확합니다.
- 서비스 레이어에서 핵심 비즈니스 검증(이메일 인증 여부, 비밀번호 정책, 본인 계정 검증, 자기 자신 팔로우 방지)을 수행합니다.
- `BusinessException + ErrorCode + GlobalExceptionHandler` 기반의 공통 에러 처리 뼈대가 있습니다.
- 단위 테스트(`MemberServiceTest`, `FollowServiceTest`)로 주요 실패 시나리오를 점검하고 있습니다.

운영 기준으로는 아래가 핵심 리스크입니다.

- 이메일 인증 우회 코드(`000000`)가 남아 있어 보안 우회 가능성이 있습니다.
- 응답 포맷이 API마다 달라 프론트/모니터링 일관성이 떨어질 수 있습니다.
- Neo4j 쿼리와 보안 설정에 대한 통합 테스트가 부족해 회귀 위험이 있습니다.
- 프로파일/시크릿/관측(로그/메트릭/감사로그) 체계가 아직 약합니다.

---

## 2) 영역별 상세 리뷰 + 추가 개선안

### 2.1 인증(Auth)

#### 👍 잘된 점
- `PasswordEncoder.matches`로 안전한 비밀번호 검증.
- 세션 저장 키가 상수로 관리되어 유지보수성이 좋음.
- Redis TTL을 활용한 인증 코드 만료 처리.

#### ⚠️ 개선 필요 (기존)
1. 이메일 인증 우회 코드 제거 필요.
2. AuthController 책임 경계 재정의 필요.
3. 로그인 실패 상태코드(400 vs 401) 정책 정리 필요.

#### ➕ 추가해야 할 점 (상세)
1. **이메일 인증 재시도/차단 정책**
   - 실패 횟수 누적(예: 5회) 시 일정 시간 차단(예: 10분).
   - Redis에 `auth:attempt:{email}` 카운터와 TTL 적용.
2. **로그인 시도 제한(Brute force 방지)**
   - IP + email 조합으로 횟수 제한.
   - 차단 시 공통 에러코드(`AUTH_429_1`) 추가.
3. **세션 보안 강화**
   - 로그인 직후 세션 재생성(session fixation 방지) 확인.
   - 쿠키 속성(`HttpOnly`, `Secure`, `SameSite`) 환경별 명시.
4. **인증 이벤트 감사로그**
   - 로그인 성공/실패, 로그아웃, 이메일 인증 성공/실패를 구조화 로그(JSON)로 기록.

---

### 2.2 보안(Security)

#### 👍 잘된 점
- `SecurityFilterChain` 기반 설정.
- `SessionAuthenticationFilter`로 세션 인증 상태를 `SecurityContext`에 주입.
- 공개/인증필수 경로를 분리하려는 의도가 코드에 반영됨.

#### ⚠️ 개선 필요 (기존)
1. 공개 API 범위 최소화.
2. 인증/인가 실패 응답 포맷 통일.
3. 권한 모델 확장성 준비.

#### ➕ 추가해야 할 점 (상세)
1. **보안 정책 표(문서) 고정**
   - 각 엔드포인트별 `PermitAll / Authenticated / Role` 매트릭스 작성.
   - 코드와 README 둘 다 유지해 변경 시 누락 방지.
2. **CORS 정책 명시**
   - 허용 Origin을 환경변수화하고 `*` 지양.
3. **보호 자원 기본 거부(Default Deny)**
   - 신규 API 추가 시 자동으로 인증 필요가 되도록 마지막 규칙 유지.
4. **보안 테스트 세트 구성**
   - 익명 접근 차단 테스트, 세션 없는 접근 테스트, 권한없는 접근 테스트를 표준화.

---

### 2.3 회원(Member)

#### 👍 잘된 점
- 회원가입 시 이메일 인증 여부 + 비밀번호 정책 함께 검증.
- Neo4j 그래프 질의(검색/경로탐색)로 도메인 적합성 높음.

#### ⚠️ 개선 필요 (기존)
1. 이메일 중복 가입 방지.
2. 공개 경로탐색 API 오남용 방지.
3. 검색 API URI 구조 안정화.

#### ➕ 추가해야 할 점 (상세)
1. **입력값 정규화**
   - email 소문자/trim 정규화 후 저장.
   - name/major 공백 정리 및 길이 제한.
2. **멤버 상태 필드 도입 검토**
   - `ACTIVE`, `SUSPENDED`, `DELETED` 상태 필드로 soft-delete 기반 운영 대응.
3. **개인정보 최소 노출 점검**
   - DTO에서 비공개 필드(이메일, bio 정책 등) 공개 범위 재검토.
4. **검색 품질 개선**
   - 부분 일치 외에 초성/유사도 검색 필요 시 별도 인덱싱 전략 고려.

---

### 2.4 팔로우(Follow)

#### 👍 잘된 점
- 요청/승인/거절 플로우 구현.
- 본인 계정 검증, 자기 자신 팔로우 방지, 중복 요청 방지.

#### ⚠️ 개선 필요 (기존)
1. `DELETE r RETURN count(r)` 영향 건수 판단 보강.
2. 동시성 레이스 보완.
3. 목록 API 페이징 도입.

#### ➕ 추가해야 할 점 (상세)
1. **요청 상태 모델 명시화**
   - 관계를 바로 삭제/생성하기보다 `PENDING/APPROVED/REJECTED` 이벤트 이력 테이블(또는 노드) 검토.
2. **알림 연계 포인트**
   - 요청/승인/거절 시점에 알림 이벤트 발행(추후 비동기 큐 연계).
3. **차단(Block) 기능 고려**
   - 팔로우 도메인과 충돌되는 정책(차단 시 기존 팔로우 제거 등) 명세 필요.
4. **관계 일관성 배치 점검**
   - 주기적으로 잘못된 관계(중복/자기참조)를 정리하는 관리성 작업 추가.

---

### 2.5 예외/응답 표준화

#### 👍 잘된 점
- 에러코드 체계가 존재하고 확장 가능.
- Validation 예외 공통 처리.

#### ⚠️ 개선 필요 (기존)
1. 성공 응답 포맷 통일 필요.
2. 예외 로깅 강화 필요.

#### ➕ 추가해야 할 점 (상세)
1. **에러 스키마 버전 필드**
   - `errorVersion` 추가로 클라이언트 호환성 관리.
2. **추적 ID(traceId) 응답 포함**
   - 장애 분석 시 요청-로그 연계가 빨라짐.
3. **도메인별 에러코드 네임스페이스 정교화**
   - 예: `FOLLOW_409_DUPLICATE_REQUEST`처럼 의미가 즉시 드러나도록 개선.
4. **성공 응답도 code 표준화**
   - `OK`, `CREATED`, `ACCEPTED` 등 상태 의미를 코드에 반영.

---

### 2.6 테스트/품질

#### 👍 잘된 점
- 서비스 단위 테스트가 핵심 도메인 검증을 커버.

#### ⚠️ 개선 필요 (기존)
1. 보안 통합 테스트 부족.
2. Neo4j 쿼리 통합 테스트 부족.
3. Redis/메일 포함 시나리오 테스트 전략 미흡.

#### ➕ 추가해야 할 점 (상세)
1. **테스트 피라미드 재구성**
   - Unit(빠름) + Slice(Repository/Web) + Integration(실환경 유사) 비율 정의.
2. **계약 테스트(Contract) 도입**
   - 프론트가 의존하는 응답 스키마를 고정.
3. **회귀 테스트 체크리스트 자동화**
   - 배포 전 필수 시나리오를 Gradle task로 묶기.
4. **성능 기준선 수립**
   - 검색/경로탐색 지연 시간의 최소 SLO(예: p95 300ms) 정의.

## 3) 추가 구현 제안 (신규 섹션)

### 3.1 반드시 추가할 기능 (기능 관점)
1. 팔로우 요청 알림 API.
2. 팔로우 요청 목록 페이징 + 정렬(최신순).
3. 사용자 상세 조회(공개 범위 제어 포함).
4. 프로필 수정 API(자기소개/인스타/전공).
5. 계정 비활성화(탈퇴) API.

### 3.2 반드시 추가할 운영 기능 (운영 관점)
1. 헬스체크 상세화(DB/Redis/메일).
2. 감사로그 조회용 관리자 엔드포인트(내부망 한정).
3. 비정상 트래픽 감지(로그인 실패 급증 알림).
4. 장애 대응 런북 문서화(메일 장애, Redis 장애, Neo4j 장애 시 조치).

### 3.3 데이터/마이그레이션 관점
1. Neo4j 인덱스/제약 조건 명시 문서 추가.
2. 버전업 시 스키마 변경 절차(마이그레이션 순서) 문서화.
3. 샘플 데이터 시드 스크립트로 로컬 재현성 개선.

## 4) 우선순위 개선 로드맵 (세분화)

### P0 (즉시: 보안/치명도)
1. 이메일 인증 우회 코드 제거(또는 local 프로필 한정).
2. 인증/인가 실패 응답 포맷 통일(EntryPoint/DeniedHandler).
3. 팔로우 쿼리 영향 건수 반환 로직 수정.
4. 공개 API 최소화 점검 + 문서화.

### P1 (단기: 안정성/일관성)
1. 이메일 중복 가입 차단(서비스 + 데이터 제약).
2. 성공 응답 포맷 전면 통일(`ApiResponse`).
3. 보안 통합 테스트 + 저장소 통합 테스트 추가.
4. 로그인/이메일 인증 시도 제한 도입.

### P2 (중기: 확장성/운영성)
1. 팔로우 요청 알림, 차단 기능, 프로필 관리 확장.
2. 운영 프로필 분리(`local/dev/prod`) + 시크릿 체계 정리.
3. 관측성(구조화 로그, traceId, 메트릭, 알림) 강화.
4. 성능/부하 테스트 및 SLO 기반 개선.

## 5) 실행 체크리스트 (바로 사용 가능)

### 보안 체크
- [ ] 개발용 인증 우회 코드 제거됨.
- [ ] 인증 필요 API가 문서와 코드에서 일치함.
- [ ] 세션 쿠키 보안 속성(`HttpOnly`, `Secure`, `SameSite`)이 환경별로 명시됨.

### API 체크
- [ ] 모든 성공 응답이 동일 스키마를 사용함.
- [ ] 모든 에러 응답이 동일 스키마 + traceId를 포함함.
- [ ] 검색/목록 API는 페이징 파라미터를 지원함.

### 데이터 체크
- [ ] 이메일 중복 불가 보장(서비스 + DB 제약).
- [ ] Neo4j 핵심 쿼리 성능/인덱스 점검 완료.

### 테스트 체크
- [ ] 서비스 단위 테스트 통과.
- [ ] 보안 통합 테스트 통과.
- [ ] 저장소 통합 테스트 통과.
- [ ] 핵심 회귀 시나리오 자동화 완료.

## 6) 결론

현재 OJK는 기능적으로 이미 좋은 출발점에 있습니다.  
다만 다음 단계에서는 기능 추가보다 **보안 우회 제거, 응답/정책 일관성, 테스트/관측 체계 강화**가 우선입니다.

이 문서의 P0/P1 항목만 먼저 완료해도, 실제 서비스 신뢰도(보안·장애 대응·개발 생산성)는 체감될 정도로 올라갈 것입니다.
